1. PROJECT OVERVIEW
================================================================================

A web-based application for managing multiple discussion groups, called Spark Groups, with features for
member management, topic tracking, event scheduling, and role-based access
control. The application supports multiple clubs, each with their own members,
topics, and events.

3. USER ROLES & PERMISSION LEVELS
================================================================================

There are four types of users: super admin, site admin, member, and pending.

3.1 PENDING USER
   - Default status for new account requests
   - Limited or no access until promoted by admin
   - Can view own profile information
   - Can edit or delete own profile

3.2 MEMBER
   - Active member with club access
   - Can belong to multiple clubs
   - Can participate in club activities
   - Can ask to join any Public club
   - Can generate a URL to invite another person to the club

3.3 CLUB ADMIN
   - Member of a club
   - Administrator of the club
   - Can approve members of the club
   - Can remove members of the club
   - Can approve topics of the club
   - Can schedule events of the club

3.3 SITE ADMIN
   - Manages users and clubs across the entire site
   - Acts as club admin for all clubs
   - Cannot modify super admin accounts

3.4 SUPER ADMIN
   - Highest privilege level
   - Can perform all site admin functions
   - Can promote/demote super admin status
   - Cannot change their own user type (requires another super admin)

IMPORTANT: System must have at least one super admin at all times.

4. AUTHENTICATION & USER REGISTRATION
================================================================================

4.1 REGISTRATION FLOW
   1. Home page displays "Request an Account" button
   2. User clicks button and is redirected to create an account
   3. User provides:
      - First name (required)
      - Last name (required)
      - Email address (required)
      - Zip code (required, 5-digit US format)
      - Bio (optional, max 500 characters, one paragraph)
   4. System validates unique email address
   5. User account created with "pending" status

4.2 EMAIL UNIQUENESS
   - No two users can share the same email address
   - System validates on registration and profile updates
   - Case-insensitive comparison

5. DATA MODELS
================================================================================

5.1 USER
   - id (primary key)
   - email (unique, indexed)
   - first_name
   - last_name
   - zip_code
   - bio (text, nullable)
   - user_type (enum: inactive, pending, member, site_admin, super_admin)
   - hashed_password
   - created_at (timestamp)
   - updated_at (timestamp)
   - last_login (timestamp)

5.2 CLUB
   - id (primary key)
   - name (unique, required)
   - description (text, short description)
   - zip_code (5-digit)
   - status (enum: public, private)
   - created_at (timestamp)
   - created_by (foreign key to user)
   - is_active (boolean, default true)

5.3 CLUB_MEMBERSHIP
   - id (primary key)
   - club_id (foreign key)
   - user_id (foreign key)
   - status (enum: pending, active, removed)
   - is_admin (boolean, indicates club admin)
   - host_order (integer, nullable, for host rotation)
   - joined_at (timestamp)
   - updated_at (timestamp)
   - created_by (foreign key to user)
   - Unique constraint on (club_id, user_id)

5.4 TOPIC
   - id (primary key)
   - club_id (foreign key)
   - title (required)
   - description (required)
   - tabs (varchar 128)
   - status (enum: pending, active, inactive, hidden)
   - created_by (foreign key to user)
   - created_at (timestamp)
   - updated_at (timestamp)
   - Unique constraint on (club_id, title) - case insensitive

5.5 TOPIC_INTEREST
   - id (primary key)
   - topic_id (foreign key)
   - user_id (foreign key)
   - interest_type (enum: interested, able_to_lead, not_interested, unspecified)
   - created_at (timestamp)
   - Unique constraint on (topic_id, user_id, interest_type)

5.6 EVENT
   - id (primary key)
   - club_id (foreign key)
   - topic_id (foreign key, nullable for non-book events)
   - title (required)
   - start_datetime (datetime, required)
   - end_datetime (datetime, required)
   - location (text, required)
   - host_id (foreign key to user)
   - status (enum: pending, active, inactive, cancelled)
   - created_by (foreign key to user)
   - created_at (timestamp)
   - updated_at (timestamp)

5.7 EVENT_ATTENDANCE
   - id (primary key)
   - event_id (foreign key)
   - user_id (foreign key)
   - rsvp_status (enum: attending, maybe, not_attending)
   - created_at (timestamp)
   - Unique constraint on (event_id, user_id)

5.7 EVENT_TOPICS
   - id (primary key)
   - event_id (foreign key)
   - topic_id (foreign key)
   - created_at (timestamp)
   - Unique constraint on (event_id, topic_id)

6. FEATURES BY USER TYPE
================================================================================

6.1 SUPER ADMIN PERMISSIONS
   All site admin permissions, PLUS:
   - Promote any user to super admin
   - Demote any super admin to site admin (except themselves)
   - Change user_type (member -> site_admin)
   - View system-wide analytics and reports
   - Access audit logs

6.2 SITE ADMIN PERMISSIONS
   All member permissions, PLUS:
   - View list of all users across all clubs
   - Change user type (pending -> member, member -> inactive, inactive -> pending )
   - Cannot modify super admin users
   - Cannot change own user type
   - Change club type (pending -> active, active -> pending, private -> public, public -> private)
   - Promote club members to club admin
   - Edit club information (name, description, zip code)
   - Deactivate/reactivate clubs
   - Act as club admin for ALL clubs

6.3 CLUB ADMIN PERMISSIONS (per club)
   All member permissions for their club, PLUS:
   - View all club members (including pending)
   - Assign users to clubs (create club membership)
   - Approve pending members (change status to active)
   - Remove members from club (status -> removed)
   - Add members to host rotation list
   - Change order of hosts in rotation
   - Change status of any topic (pending -> active, etc.)
   - View detailed topic interest (names of interested/leading/not interested/unspecified)
   - Create events (initial status: pending)
   - Edit event details (title, time, location, host, topic list)
   - Change event status (pending -> active -> inactive/cancelled)
   - Remove members from events
   - View all event attendees

6.4 MEMBER PERMISSIONS
   - View own profile, edit bio and zip code, change password
   - Browse list of all public clubs (with search/filter)
   - Create new book clubs (with Pending status)
   - Request to join a public club (creates pending membership)
   - Leave any club they belong to
   - Belong to multiple clubs simultaneously
   - View their personal list of topics
   - Delete and add to their personal list of topics
   
   Within clubs they're an active member of:
   - View list of club members (active only) and club admins
   - View member profiles (name, bio, if member opts in)
   - View topics with status: active, inactive (not pending or hidden)
   - View topic details (title, tags, description)
   - View aggregate counts (how many interested/leading/not interested/unspecified)
   - Create new topics (initial status: pending)
   - Mark themselves as interested/leading/not interested/unspecified for any topic
   - Remove themselves from interested/leading/not interested/unspecified list of any topic
   - Sort topic list by any field (title, tags, status, created date)
   - Search topics by title or tags
   - View active events
   - RSVP to future events (add/remove themselves)
   - View event details (topics, time, location, host, attendance count)
   - Generate a URL to send to someone to invite them to join the club (may require the person to create a user)

6.5 PENDING USER PERMISSIONS
   - View own profile
   - Edit profile information
   - Wait for approval from admin
   - Delete own profile

7. BUSINESS RULES & EDGE CASES
================================================================================

7.1 USER MANAGEMENT
   - System must maintain at least one super admin at all times
   - Users cannot change their own user type
   - Removed club members retain topics they created in that club
   - User deletion should be soft delete (retain for audit trail)
   - Email addresses are case-insensitive and trimmed

7.2 CLUB MANAGEMENT
   - Each club must have at least one club admin
   - Club names must be unique across the site
   - Zip codes used for suggesting nearby clubs to users
   - Host rotation is optional; clubs can manage manually
   - Inactive clubs are hidden from browse list but data retained
   - Newly created clubs are private until a site admin or club admin changes them to public

7.3 TOPIC MANAGEMENT
   - Topic titles must be unique within a club (case-insensitive)
   - Same topic title can exist in different clubs
   - Topics with pending status only visible to club admins
   - Hidden topics not visible to members, only admins
   - Users can have only one interest type for same topic simultaneously
   - Topic creator remains associated even if removed from club

7.4 EVENT MANAGEMENT
   - Events can be associated with a set of topics 
   - Events without topics are not allowed 
   - Only future events can be RSVP'd to
   - Past events are read-only (for history)
   - Super admins can delete a past event
   - Host must be an active member of the club
   - Event times validated (end must be after start)
   - Cancelled events remain visible but cannot be RSVP'd to

7.5 HOST ROTATION
   - Host order is a number (1, 2, 3, etc.)
   - Multiple members can have same order (ties handled by admin)
   - Not all members need to be in rotation
   - System can suggest next host based on rotation

9.5 USABILITY

   - Accessibility compliance (WCAG 2.1 Level AA)

   - Mobile-responsive design

9.6 MAINTAINABILITY
